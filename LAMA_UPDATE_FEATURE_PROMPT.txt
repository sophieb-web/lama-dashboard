SECTION 1: WHAT THIS FEATURE DOES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A news monitor that automatically checks reliable sources every Monday
for new Israeli cyber funding news, then stages the findings for human
review before anything touches the live database.

The full flow:

  TRIGGER (Monday auto OR "Update Now" button)
       ↓
  SCRAPER runs — checks CTech, TechCrunch, Geektime, SecurityWeek,
  Crunchbase for Israeli cyber news from the last 7 days
       ↓
  STAGING — findings written to staging.json as pending changes
       ↓
  REVIEW — user sees each finding as a card, approves or rejects
       ↓
  PUSH — approved changes merged into deals.csv + taxonomy.csv,
  committed to GitHub, Railway redeploys automatically

Nothing ever goes into the live database without explicit human approval.

Sources to scrape (in this order):

2. TechCrunch (techcrunch.com)

3. Geektime (geektime.com)
   - Search URL: https://www.geektime.com/?s=cybersecurity+funding+israel
   - Filter to last 7 days
   - Extract: headline, company name, amount, investors, date, article URL

4. SecurityWeek (securityweek.com)
   - Search URL: https://www.securityweek.com/?s=israel+funding
   - Filter to last 7 days
   - Extract: headline, company name, amount, investors, date, article URL

5. Crunchbase (crunchbase.com) — for companies already in our database
   - For each company in deals.csv that has a Crunchbase URL or known name,
     check if there are new funding rounds not yet in our database
   - Only check companies where the latest round in our DB is > 6 months old
   - Note: Crunchbase may block scraping — if so, skip gracefully and log it

Scraping rules:
  - Use requests with a realistic User-Agent header
  - Add 2-3 second delays between requests to be polite
  - Wrap every request in try/except — if a source fails, log it and continue
  - Never crash the whole scrape because one source is down
  - Timeout each request after 10 seconds

What to extract per finding:
  {
    "id": "unique string (source + date + company name hashed)",
    "type": "new_round" | "new_company" | "acquisition" | "enrichment",
    "status": "pending",  // pending, approved, rejected
    "company_name": "string",
    "company_in_db": true/false,  // whether company exists in deals.csv
    "source_url": "string",
    "source_name": "CTech" | "TechCrunch" | "Geektime" | "SecurityWeek",
    "headline": "string",
    "scraped_at": "ISO timestamp",
    "data": {
      // For new_round:
      "round_type": "Seed" | "Series A" | etc,
      "round_date": "string",
      "round_size": "float or null",
      "lead_investor": "string",
      "co_investors": "string",
      // For new_company:
      "description": "string",
      "sector_tag": "string (best guess from context)",
      "hq": "string",
      "founded": "string",
      // For acquisition:
      "acquirer": "string",
      "exit_size": "string"
    }
  }

Cross-reference logic:
  - After scraping, compare company_name against all Company Name values
    in deals.csv
  - Use fuzzy matching (fuzz.ratio from rapidfuzz library) with threshold 85
    to catch "Jazz" matching "Jazz Security" etc.
  - If match found: set company_in_db = true, link to existing record
  - If no match: set company_in_db = false (new company to add)
  - Add rapidfuzz to requirements.txt

Deduplication:
  - Before adding to staging.json, check if a finding with the same
    company_name + round_type + round_date already exists in staging.json
  - If duplicate found, skip it
  - Also check against deals.csv — if the round already exists in the
    database, skip it

merger.py takes approved findings from staging.json and merges them
into deals.csv. It does NOT touch the Excel file — only the CSVs.

merge_approved() function:
  1. Load deals.csv into a pandas DataFrame
  2. Get all approved findings from staging.py
  3. For each approved finding:

     If type == "new_round" and company_in_db == true:
       - Find all existing rows for this company in the DataFrame
       - Get the company profile from the first row (cols 1-20)
       - Create a new row with that same profile + the new deal data
       - Append the new row to the DataFrame

     If type == "new_company":
       - Create a new row with whatever data was scraped
       - Fill in: Company Name, Website, Description, HQ, Sector Tag,
         Founding Year (if found), Round Type, Round Date, Round Size,
         Lead Investor, Co-Investors, Source (the article URL), Notes
       - Leave other fields blank — they'll be filled in during manual
         enrichment sessions
       - Append to DataFrame

     If type == "acquisition":
       - Find all existing rows for this company
       - Update Acquired? to "Yes", Acquirer to the acquirer name,
         Exit Size to the exit size if known

  4. Sort the DataFrame: Lama portfolio companies first, then alphabetical
     by Company Name
  5. Save back to deals.csv
  6. Move merged findings from approved → pushed in staging.json
  7. Return a summary: {merged: N, new_companies: N, new_rounds: N}

pusher.py handles the git push after merger.py runs.

Use the gitpython library (already in requirements.txt).

push_to_github() function:
  1. Get the repo root path (two levels up from lama_dashboard/)
  2. Open the git repo: repo = git.Repo(repo_root)
  3. Stage the changed files:
       repo.index.add(['lama_dashboard/data/deals.csv',
                       'lama_dashboard/data/taxonomy.csv',
                       'lama_dashboard/data/staging.json'])
  4. Create a commit:
       commit_message = f"auto-update: {N} new deals, {N} new companies — {date}"
       repo.index.commit(commit_message)
  5. Push to origin main:
       origin = repo.remote(name='origin')
       origin.push()
  6. Return success/failure status

Git authentication on Railway:
  - Railway needs a GitHub Personal Access Token to push
  - Add a new Railway environment variable: GITHUB_TOKEN = ghp_xxxxxx
  - In pusher.py, construct the remote URL with the token:
      token = os.environ.get('GITHUB_TOKEN')
      remote_url = f"https://{token}@github.com/sophieb-web/lama-dashboard.git"
      origin.set_url(remote_url)
  - Never hardcode the token in the code

To get a GitHub Personal Access Token:
  github.com → Settings → Developer Settings → Personal Access Tokens
  → Tokens (classic) → Generate new token
  → Scopes: check "repo" (full repo access)
  → Copy and add to Railway Variables as GITHUB_TOKEN

     Returns immediately: {"status": "started"}
     Page polls /api/staging to track completion
     On completion: staging.json updated with pushed findings
