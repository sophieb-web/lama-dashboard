SECTION 1: WHAT THIS FEATURE DOES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

A proactive market intelligence feed that automatically detects signals
relevant to Lama Partners — new CISO appointments, founder vesting
completions, fund activity changes, AI transformation announcements,
customer hiring surges, competitor moves, and more.

Runs automatically every day at 7am Israel time.
Roy opens his laptop and sees a prioritized briefing — what happened
overnight that Lama needs to act on today.

The full flow:

  TRIGGER (7am daily auto OR "Run Now" button)
       ↓
  SIGNAL DETECTOR runs — checks all detectable sources:
  - RSS feeds (news, CISO appointments, AI transformation)
  - Our own database (founder vesting, fund quiet periods)
  - LinkedIn Jobs scraping (hiring surges)
  - Israeli Companies Registrar (new company registrations)
       ↓
  SIGNALS WRITTEN to signals.json (all findings, prioritized)
       ↓
  SIGNALS PAGE shows prioritized daily briefing
  (Critical → High → Medium tiers)
       ↓
  User can: Dismiss | Save to watchlist | Mark action taken
       ↓
  Saved signals persist in signals.json watchlist

SECTION 4: SIGNAL_DETECTOR.PY — DETAILED SPEC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

signal_detector.py runs all signal detection logic. It is pure Python —
no AI, no Anthropic API, no charges. Just HTTP requests, RSS parsing,
and database logic.

Main function: run_all_detectors()
  - Runs every detector listed below
  - Each detector returns a list of signal dicts
  - All signals combined, deduplicated, prioritized
  - Written to signals.json
  - Returns summary: {total: N, critical: N, high: N, medium: N}

Wrap every detector in try/except — if one fails, log it and continue.
Never crash the whole run because one detector fails.

─────────────────────────────────────────────────────────────────────────────
DETECTOR 2: DATABASE CALCULATIONS
─────────────────────────────────────────────────────────────────────────────

Run these calculations against deals.csv. These are the most reliable
signals because they come from our own verified data.

CALCULATION A — FOUNDER VESTING COMPLETE (Critical priority):
  For every company in deals.csv where Acquired? == "Yes":
    Parse the Round Date of the acquisition round (or use Notes field)
    Calculate: acquisition_date + 3 years = vesting_start_window
    Calculate: acquisition_date + 4 years = vesting_end_window
    If today is within that window:
      signal_type = "Founder vesting complete"
      title = f"{founders} ({company}) — vesting window open"
      why_it_matters = "Founder likely free from vesting and considering
                        next company — prime time to build relationship"
      action = "Reach out to founder before they start taking meetings"
      priority = "Critical"
    If today is within 3 months of window opening:
      Same signal but priority = "High" (heads up — approaching)

CALCULATION B — FUND QUIET PERIOD (High priority):
  For each unique investor that appears in Lead Investor column:
    Find all deals where they appear as Lead Investor
    Find the most recent deal date
    Calculate months since most recent deal
    If months >= 6:
      signal_type = "Fund goes quiet 6+ months"
      title = f"{fund_name} — no new deals in {months} months"
      why_it_matters = "Likely raising new fund — will re-emerge with
                        fresh capital in 3-6 months"
      action = "Build relationship with fund before they re-emerge active"
      priority = "High"
    If months >= 9:
      priority = "Critical" — they are definitely raising

CALCULATION C — SERIAL FOUNDER 4-YEAR MARK (High priority):
  For every company where Acquired? == "Yes" and Founders is not empty:
    If acquisition was exactly 3.5-4.5 years ago:
      signal_type = "Serial founder's last company crosses 4-year mark"
      title = f"{founders} — 4 years post-{acquirer} acquisition"
      why_it_matters = "Serial founders typically start next company after
                        4 years at acquirer — pattern signal"
      action = "Proactive outreach — be first call when they're ready"
      priority = "High"

CALCULATION D — NON-COMPETE EXPIRY (High priority):
  Same logic as above but at 18 months post-acquisition
  signal_type = "Founder's non-compete expires"
  title = f"{founders} — non-compete window closing at {company}"
  why_it_matters = "Post-acquisition non-competes typically 12-24 months
                    — expiry means founder is legally free to compete"
  action = "Reach out as non-compete window closes"
  priority = "High"

CALCULATION E — COMPETITOR RECENTLY FUNDED (High priority):
  For Lama portfolio companies (Terra, Orion Security, Root, Capsule, Jit):
    Identify their sector tag from deals.csv
    Find all other companies in the same sector
    Find any that have a round date in the last 30 days
    signal_type = "Competitor to portfolio company raises big round"
    title = f"{competitor} raises {amount} in {portfolio_company}'s sector"
    why_it_matters = "Direct competitive threat — competitor now has runway"
    action = "Alert founder immediately — may accelerate hiring or product"
    priority = "Critical"
    portfolio_relevant = True
    portfolio_companies = [the relevant portfolio company]

DETECTOR 3: LINKEDIN JOBS MONITOR
─────────────────────────────────────────────────────────────────────────────

Scrape LinkedIn Jobs search results for security hiring surges.
Note: LinkedIn blocks scrapers aggressively. Use these strategies:
  - Rotate User-Agent headers
  - Add 3-5 second delays between requests
  - Use requests-html or playwright if basic requests fails
  - If blocked completely: log the failure and skip gracefully

Search URLs to check:
  Security roles surge:
  https://www.linkedin.com/jobs/search/?keywords=CISO&location=&f_TPR=r604800
  (f_TPR=r604800 = posted in last 7 days)

  Also check:
  https://www.linkedin.com/jobs/search/?keywords=chief+information+security+officer&f_TPR=r604800

For each job posting found:
  Extract: company name, job title, location, posted date
  If job title contains CISO / Chief Security Officer / Chief Information Security Officer:
    signal_type = "New CISO appointed" (or "CISO role open — buying window coming")
    Cross-reference company against deals.csv customers column
    If customer match: priority = "Critical"
    Else: priority = "High"

For hiring surge detection:
  This requires checking a specific company's job postings count
  Build a secondary function: check_company_hiring_surge(company_name)
    - Search LinkedIn Jobs for company_name + security
    - If 5+ security job postings found: flag as hiring surge
  Run this function for the top 50 companies in deals.csv by total raised
  (most likely to have visible LinkedIn job postings)

If LinkedIn scraping is completely blocked:
  Log: "LinkedIn scraping blocked — skipping jobs detector"
  Continue with other detectors
  Do NOT crash

SECTION 5: SIGNALS_STORE.PY — DETAILED SPEC
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

signals_store.py manages reading and writing signals.json.

Functions:
  load_signals() → dict
  save_signals(data) → None
  add_signals(signal_list) → int  (returns count of new signals added)
  dismiss_signal(signal_id) → None
  save_signal(signal_id) → None       (move to watchlist)
  mark_action_taken(signal_id, note) → None
  get_active_signals() → list         (status == "active", sorted by priority)
  get_watchlist() → list              (status == "saved")
  get_stats() → dict                  (counts by priority and category)
  get_run_status() → dict             (last_run, next_run, run_status, run_log)

Priority sort order for get_active_signals():
  Critical first, then High, then Medium
  Within same priority: newest first (by detected_at)

SECTION 10: IMPORTANT NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- Signal detection uses zero AI. No Anthropic API calls. No charges.
  All detection is keyword matching, RSS parsing, and database logic.

- signals.json is committed to GitHub as part of the daily push from
  the Update Center. This means signal state persists across Railway
  redeploys.

- The Lama portfolio companies (Terra, Orion Security, Root, Capsule,
  Jit) must always be checked for relevance in every signal. If a
  signal is relevant to a portfolio company, flag it and surface it
  first regardless of priority tier.

- Dismissed signals are stored permanently in dismissed[] and never
  re-surfaced. This prevents Roy from seeing the same signal twice.

- The database calculations (vesting, fund quiet, etc.) run on every
  detection cycle. They may produce the same signals repeatedly.
  Deduplication by (signal_type + entity + month) prevents this —
  vesting signals appear once per month maximum.

- LinkedIn Jobs scraping will likely be blocked. Build it to fail
  gracefully — log the failure, skip, and continue with other detectors.
  The page should not show an error if LinkedIn fails.

- Globes English (en.globes.co.il/en/rss) is the most important Israeli
  source for local cyber news. It often breaks Israeli cyber stories
  12-24 hours before TechCrunch picks them up.

- The run log should be detailed enough that Roy can see exactly what
  was checked and why each signal was or wasn't generated. Transparency
  builds trust in the system.
